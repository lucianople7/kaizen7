# K7 Hive Memory

## Mission

Lograr mas acierto con menos pasos y menos tokens.

K7 Hive Memory no es una base de datos nueva. Es una regla operativa para que KAIZEN7 use Obsidian como memoria canónica, consulte solo los nodos relevantes y convierta aprendizaje repetido en skill, SOP o decision.

```text
Headroom = comprimir ruido
Graphify = ver relaciones
Ponytail = accion minima
Obsidian² = memoria canonica + memoria operativa conectada
```

## Core Loop

Usar siempre el camino corto:

```text
intencion -> nodo canonico -> vecinos utiles -> decision -> accion minima -> memoria util
```

Si un paso no cambia la decision, saltarlo.

## Hive Cells

Pensar la memoria como celdas especializadas:

- `Canon`: arquitectura, decisiones firmes y reglas.
- `Daily`: movimientos recientes y bloqueos activos.
- `Domain`: THE FOCUX, Flowmatik, ecommerce, contenido, proveedores, compliance.
- `Genome`: datos estructurados de producto, creatividad, economia, suppliers y claims.
- `Evolution`: repos, herramientas, patrones externos y descartes.
- `Skills`: aprendizajes convertidos en accion reutilizable.

Obsidian manda. Las celdas observan; el canon decide.

## Read Policy

Antes de leer muchos archivos, responder:

```text
Que decision necesito tomar?
Que nodo canonico manda?
Que vecino puede cambiar la decision?
Que puedo no leer?
Que memoria debo actualizar al cerrar?
```

Escalera:

1. Intencion del usuario.
2. Nota canonica o indice.
3. Semaforo/daily si importa estado reciente.
4. Celda de dominio.
5. `rg` o mapa/grafo.
6. Archivos concretos.
7. Herramientas externas solo si el dolor es real.

## Write Policy

Guardar solo lo que evita repetir trabajo:

- decision nueva,
- decision actualizada,
- error que no debe repetirse,
- patron reutilizable,
- bloqueo activo,
- siguiente accion concreta.

No guardar:

- resumenes decorativos,
- outputs largos,
- secretos,
- datos personales innecesarios,
- hipotesis como si fueran decisiones.

## Conflict Rule

Si dos memorias contradicen:

1. Priorizar arquitectura canonica.
2. Luego decision ledger o nota de dominio mas reciente.
3. Luego daily.
4. Luego inferencia del agente.

Si la contradiccion afecta trabajo futuro, escribir:

```text
DECISION ACTUALIZADA
Fecha:
Que cambia:
Que invalida:
Ruta afectada:
Siguiente accion:
```

## Ponytail Gate

Antes de crear una nota, skill, script, indice o grafo nuevo:

```text
Existe ya?
Reduce pasos o tokens hoy?
Evita repetir un error?
Cabe en una pieza pequena?
```

Si no cumple 2 de 4, no crear.

## Graph Mode

Usar grafo mental primero:

```text
nodo actual -> vecinos directos -> decision
```

Solo crear grafo tecnico o instalar herramienta si:

- la exploracion grande se repite,
- `rg` + Obsidian no bastan,
- hay dudas recurrentes de arquitectura,
- o el coste de contexto empieza a doler.

## Output Format

Cuando esta skill guie una tarea, cerrar con:

```text
Nodo canonico:
Vecinos leidos:
Lecturas evitadas:
Decision:
Accion minima:
Memoria actualizada:
```
