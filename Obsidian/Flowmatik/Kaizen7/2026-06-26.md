## Signal Ingestion v0

### Fecha

2026-06-26

### Decision

KAIZEN7 incorpora `Signal Ingestion v0` como capa de entrada para sacar patrones utiles de GitHub, Hugging Face, web, ecommerce, proveedores, OCR o texto libre sin cargar contexto completo.

### Implementado

- `lib/signal-ingestion.js`: normaliza fuente, limpia Markdown, extrae claims, entidades, precios, ingredientes, herramientas, riesgos y senales de mercado.
- `data/signal-inbox.json`: inbox append-only para paquetes compactos.
- `npm.cmd run k7:signal`: CLI para convertir una fuente en paquete accionable.
- `production-readiness`: ahora exige signal ingestion, inbox y script `k7:signal`.
- Tests: `tests/signal-ingestion.test.js` y readiness actualizado.

### Criterio

Adoptar el patron comun de las mejores herramientas modernas: entrada modular, metadatos primero, scoring minimo, siguiente accion concreta y lectura profunda solo si la senal merece ejecucion.

### Siguiente accion

Construir adapters v1: GitHub metadata, Hugging Face Hub metadata y web Markdown extractor opcional. Mantenerlos separados del core hasta demostrar menos pasos, menos tokens o mejor decision.

## GitHub Adapter v1

### Fecha

2026-06-26

### Decision

KAIZEN7 incorpora `GitHub Adapter v1` como primer adaptador real de mercado para convertir repositorios en paquetes `Signal Ingestion` accionables.

### Implementado

- `lib/github-adapter.js`: parsea URLs de repos, consulta contrato GitHub, construye texto de senal, puntua licencia/adopcion/actividad/ecosistema y genera un `SignalPacket`.
- `npm.cmd run k7:github -- "https://github.com/org/repo"`: CLI para evaluar repos.
- `--write`: guarda el paquete en `data/signal-inbox.json`.
- `production-readiness`: ahora exige modulo y script `k7:github`.
- Tests: `tests/github-adapter.test.js` y readiness actualizado.

### Criterio

El adaptador no instala repos ni copia herramientas completas. Extrae patron, licencia, madurez y utilidad real; si hay riesgo de licencia, lo manda a decision antes de implementacion.

### Siguiente accion

Conectar `Hunter` para leer `data/signal-inbox.json` y priorizar automaticamente las senales GitHub con score alto.

## Agent Runner v1

### Fecha

2026-06-26

### Decision

KAIZEN7 incorpora `Agent Runner v1` como interfaz principal de uso real para agentes y humanos. El objetivo es dejar de operar con comandos sueltos y exponer una entrada unica de produccion.

### Implementado

- `lib/agent-runner.js`: compone objetivo, ingestion opcional GitHub/HF, memoria, skills, Signal Inbox, Hunter queue y una accion priorizada.
- `npm.cmd run k7:run -- "objetivo"`: modo humano.
- `npm.cmd run k7:run -- --compact "objetivo"`: contrato compacto para otros agentes.
- `npm.cmd run k7:run -- --json "objetivo"`: contrato completo.
- `--github` y `--hf`: ingestan fuentes reales y las guardan en `data/signal-inbox.json` antes de decidir.
- `production-readiness`: ahora exige modulo y script `k7:run`.
- Tests: `tests/agent-runner.test.js` y readiness actualizado.

### Regla

`k7:run` devuelve una sola accion priorizada. La politica es single-action first, metadata first y lectura profunda solo del candidato, skill o memoria seleccionada.

### Siguiente accion

Convertir `k7:run` en endpoint HTTP local para que cualquier agente externo lo pueda llamar sin shell.

## KAIZEN7 Local API v1

### Fecha

2026-06-26

### Decision

KAIZEN7 expone `Agent Runner v1` como API local en `POST /api/k7/run`. Esto convierte el sistema en una herramienta conectable por agentes, apps o interfaces sin depender de shell.

### Implementado

- `server.js`: endpoint `POST /api/k7/run`.
- Request soporta `goal`, `githubUrls`, `huggingFaceUrls`, `github`, `hf` y `compact`.
- Respuesta completa devuelve el run completo; `compact: true` devuelve resumen estable para agentes.
- `tests/server.integration.test.js`: cubre el endpoint desde servidor real.
- `production-readiness`: ahora verifica que `/api/k7/run` existe.

### Contrato

```json
{
  "goal": "objective",
  "githubUrls": ["https://github.com/org/repo"],
  "huggingFaceUrls": ["https://huggingface.co/BAAI/bge-m3"],
  "compact": true
}
```

### Siguiente accion

Crear una pequena pantalla/API client en la WebUI para llamar `POST /api/k7/run` visualmente y usarlo como consola de produccion.

## Agent Advisor v1

### Fecha

2026-06-26

### Decision

KAIZEN7 incorpora `Agent Advisor v1` como capa que cualquier agente externo querria consultar antes de actuar. No ejecuta; aconseja como trabajar mejor con menos contexto, menos riesgo y menos tokens.

### Implementado

- `lib/agent-advisor.js`: genera advice para agentes externos segun objetivo, capacidades, presupuesto de contexto y tolerancia al riesgo.
- `npm.cmd run k7:advise`: CLI para Codex, Hermes, Mastra, OpenClaw u otros agentes.
- `POST /api/k7/advise`: endpoint local para integracion HTTP.
- `server.js`: expone el endpoint.
- `production-readiness`: exige modulo, script y API.
- Tests: `tests/agent-advisor.test.js`, `tests/server.integration.test.js` y readiness actualizado.

### Contrato

```json
{
  "agent": "codex",
  "goal": "fix failing tests and improve routing",
  "capabilities": ["read_files", "edit_files", "run_tests"],
  "contextBudget": 1200,
  "riskTolerance": "low",
  "compact": true
}
```

### Salida

- memoria minima que leer,
- skills exactas,
- riesgos/acciones a evitar,
- primera accion segura,
- politica de tokens,
- comandos de verificacion.

### Siguiente accion

Crear un `KAIZEN7 MCP Server v1` para exponer `k7_advise` y `k7_run` como tools nativas para agentes compatibles con MCP.
